## Stale-CAPTURING Conv Bug — Lesson 2026-06-05 12:48 CT

**Bug:** The cron's `in_window()` function passed CAPTURING conversations unconditionally. A work-call conversation (8496985) that started at 11:01 and ended at 11:58 was still flagged CAPTURING in Bee's data at 12:29 — the 12:29 cron run pulled it, treated it as "in the last 30 min," and surfaced the rash mention as a current finding. Chris caught it: "I did not mention the rash in the last 30 minutes so that's false in the last cron output. It's also still talking about my work call so is not only looking at the last 30 minutes. The work call ended at 11."

**Root cause:** Bee takes a long time to finalize conversations — a 1-hour work call that ended at 11:58 was still CAPTURING in Bee's metadata at 12:29, 50+ minutes later. The cron's `in_window()` only checked `state == "CAPTURING"` and passed it without any time check. Result: stale CAPTURING convs from hours ago were treated as "in the last 30 min."

**Fix applied at 12:48 CT:** `bee-live-window.py`'s `in_window()` now enforces a freshness check on CAPTURING convs:
  - A CAPTURING conv passes only if its latest `spoken_at` (across all utterances) is within the last `2 * window_minutes` minutes (60 min for a 30-min window)
  - OR if it has no utterances yet AND its start_time is within the last `2 * window_minutes` minutes
  - Stale CAPTURING convs (>60 min since last utterance) are excluded, even though Bee hasn't finalized them

**Verified at 12:53 CT:** With the same data, 8496985 (work call, latest_utt 11:51, 63 min old) is now correctly EXCLUDED. 8498598 (milkshake, CAPTURING, latest_utt 12:53) is correctly INCLUDED.

**Cron prompt updated** to include a new section: '"LAST 30 MINUTES" MEANS LAST 30 MINUTES — STALE CAPTURING CONVS ARE EXCLUDED' with the rule, the rationale, and the directive to trust the script's freshness check.
